You now have save points. Branches are the next unlock: a way to work on something risky
*without* touching the version that already works.

## A parallel universe for your code

A **branch** is a separate line of history. Picture your commits as a timeline. A branch
splits off a parallel copy where you can make changes freely. The original timeline
(usually called `main`) stays exactly as it was.

If the experiment works out, you **merge** it back into `main`. If it doesn't, you throw the
branch away and `main` never even knew it happened.

<Callout type="tip" title="Why you'd want this">
  Your live website runs off `main`. You want to try a big redesign. Do it on a branch — your
  real site stays safe on `main` while you experiment. Merge only when the redesign is ready.
</Callout>

## The commands

Create a branch and switch to it in one step:

<Terminal cmd="git switch -c redesign-homepage" out={`Switched to a new branch 'redesign-homepage'`} />

Now every commit you make lands on `redesign-homepage`, not `main`. Make your changes,
commit them as usual. When you want to go back to the safe version:

<Terminal cmd="git switch main" out={`Switched to branch 'main'`} />

Your files instantly change back to how `main` looks. The redesign work isn't gone — it's
safely on its branch, waiting. Switch back anytime with `git switch redesign-homepage`.

## Merging it back

Happy with the branch? Bring it into `main`. First switch to the branch you want to merge
*into* (`main`), then merge the other one in:

<Terminal cmd="git switch main" out={`Switched to branch 'main'`} />

<Terminal cmd="git merge redesign-homepage" out={`Updating a1b2c3d..f4e5d6c
Fast-forward
 index.html | 40 ++++++++++++++++++++++++++--------------`} />

Now `main` contains the redesign. You can delete the branch — it's served its purpose.

<Callout type="warning" title="You'll hear the word 'conflict'">
  Sometimes two branches change the same line and git can't decide which to keep. That's a
  **merge conflict**. It sounds scary but it's just git asking you to choose. We cover
  handling them calmly in a later lesson — for now, know that it's normal and fixable.
</Callout>

<TryItInRepo>
  In a git project you've got lying around:

  1. Run `git switch -c my-experiment` to branch off.
  2. Change a file and commit it (`git add .` then `git commit -m "Try something"`).
  3. Run `git switch main` — watch your change vanish from the files.
  4. Run `git switch my-experiment` — watch it come back.

  Feel that? That's the safety net. Nothing you do on a branch can hurt `main`.
</TryItInRepo>
